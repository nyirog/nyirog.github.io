Discover dbus

Published: Tue 05 January 2021
By Nyirő, Gergő

In dbus.

I'm no expert in this subject by any means just read some tutorial about dbus so I thought I could summarize them for other beginners like me.

List connections

Let see who is listening on the session bus:

$ dbus-send --session --dest=org.freedesktop.DBus --type=method_call --print-reply \
  /org/freedesktop/DBus org.freedesktop.DBus.ListNames
   array [
      string "org.freedesktop.DBus"
      string ":1.403"
      string "org.freedesktop.Notifications"
      string ":1.405"
      string "org.freedesktop.portal.Desktop"
      string "org.freedesktop.systemd1"
      string ":1.408"
      string "org.pulseaudio.Server"
      ...
   ]

This command requires some explanations. dbus-send call the ListNames method of /org/freedesktop/DBus object at the org.freedesktop.DBus address. The ListNames method is defined in the org.freedesktop.DBus interface. The org.freedesktop.DBus seems quite repetative.

The reply is the list of bus names on the session bus. You can see two kind of bus names: unique name (:1.403) and well known name (org.pulseaudio.Server). All the bus clients has a uniq name but they can require well known name, too. System services and managed services usually has well known name for easier access.

There are two reasons why org.freedesktop.DBus is so repetative in our example: unique names and global interfaces.

Address

Dbus use the unique name in the protocol the well known name is only a helping mechanism for the coordination of dbus services.

Object

However the name of the object can be used in match filters so it should be uniq.

Interface

The interfaces meant to be global and reusable therefore they should be uniq too. There standard interfaces which are implemented by all of the client.
